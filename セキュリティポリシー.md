# セキュリティポリシー

**バージョン**: 1.0  
**作成日**: 2026-04-11  
**適用プロジェクト**: マインドマップツール（全フェーズ）

---

## 1. 方針

本プロジェクトはユーザーの情報資産を守ることを最優先とします。
セキュリティは後付けではなく、設計・実装・運用のすべての段階で考慮します。

---

## 2. 脅威モデル（フェーズ別）

### Phase 1（サーバーレス・認証なし）

| 脅威 | リスク | 対策 |
|------|--------|------|
| XSS（クロスサイトスクリプティング） | 中 | テキストのエスケープ処理 |
| 悪意あるファイルインポート | 中 | JSON/テキストバリデーション |
| ローカルデータの漏洩 | 低 | 機密情報を入力しないよう利用ガイドで注意喚起 |
| 悪意ある JS ライブラリ（サプライチェーン） | 中 | npm audit・バージョン固定 |

### Phase 2 以降（認証・クラウド保存）

| 脅威 | リスク | 対策 |
|------|--------|------|
| 認証情報の漏洩 | 高 | HTTPS 必須・トークン適切管理 |
| 不正アクセス（認可不備） | 高 | リソース所有者チェック |
| CSRF | 中 | SameSite Cookie・CSRF トークン |
| ブルートフォース | 中 | レートリミット |

## 3. フロントエンドセキュリティ規約

### 3.1 XSS 対策（必須）

ユーザー入力テキスト（ノードのテキスト等）は**必ずエスケープ処理して描画**する。
`dangerouslySetInnerHTML` の使用を原則禁止する。

```typescript
// 禁止
<div dangerouslySetInnerHTML={{ __html: userInput }} />

// 推奨（React はデフォルトでエスケープする）
<div>{userInput}</div>
```

`innerHTML` へのユーザー文字列の直接代入も禁止。

```typescript
// 禁止
element.innerHTML = userInput;

// 推奨
element.textContent = userInput;
```

### 3.2 ファイルインポートのバリデーション（必須）

JSON・テキストファイルのインポート時は以下を検証する。

**JSONインポート**

```typescript
// 検証項目
- ファイルサイズが上限（5MB）以内か
- JSON としてパースできるか
- スキーマバージョンが対応範囲内か
- 必須フィールド（root, id, text, children）が存在するか
- children が配列であるか
- text フィールドが文字列であり、長さが上限（500文字）以内か
- ネスト深さが上限（50階層）以内か（無限ループ防止）
```

**テキストインポート**

```typescript
// 検証項目
- ファイルサイズが上限（1MB）以内か
- 行数が上限（10,000行）以内か
- インデント深さが上限（50階層）以内か
```

不正データを検知した場合は処理を中断し、ユーザーにエラーメッセージを表示する。

### 3.3 コンテンツセキュリティポリシー（CSP）

`index.html` に CSP メタタグを設定する。

```html
<meta http-equiv="Content-Security-Policy"
  content="default-src 'self';
           script-src 'self';
           style-src 'self' 'unsafe-inline';
           img-src 'self' blob: data:;
           font-src 'self';
           connect-src 'self';
           object-src 'none';
           base-uri 'self';">
```

> `style-src` の `'unsafe-inline'` は Tailwind CSS のために許可。
> Phase 2 以降でインラインスタイルが不要になれば除去する。

### 3.4 localStorage の取り扱い

- localStorage に保存するのはマップデータのみとする
- パスワード・認証トークン等を localStorage に保存しない（Phase 2 以降）
- 個人情報・機密情報をマップに記載することをユーザーに注意喚起する

## 4. 依存ライブラリセキュリティ

### 4.1 定期チェック

```bash
# 週次で実行することを推奨
npm audit

# 自動修正（minor/patch のみ）
npm audit fix
```

### 4.2 ライブラリ追加ルール

- メジャーなエコシステムに属するライブラリを優先する
- 週間ダウンロード数が 10,000 未満の無名ライブラリは原則使用しない
- 追加前に `npm audit` を実行し、既知の脆弱性がないか確認する
- バージョンは `package.json` でキャレット（`^`）で固定し、意図しないアップデートを防ぐ

### 4.3 脆弱性が発見された場合

1. `npm audit fix` で自動修正を試みる
2. 自動修正できない場合は代替ライブラリを検討する
3. 修正内容を devlog に記録する

## 5. ビルド・デプロイセキュリティ

### 5.1 ビルド成果物

- 本番ビルド（`npm run build`）では `console.log` がないことを確認する
- ソースマップは本番環境に含めない（`.gitignore` で `dist/` 管理）
- 環境変数（`.env` ファイル）は git にコミットしない

### 5.2 デプロイ先の設定

| 設定 | 要件 |
|------|------|
| HTTPS | 必須（Phase 2 以降）。Phase 1 のローカル利用を除く |
| キャッシュ制御 | `index.html`: `no-cache`、アセット: 長期キャッシュ |
| X-Content-Type-Options | `nosniff` |
| X-Frame-Options | `DENY` または `SAMEORIGIN` |

## 6. インシデント対応

### 6.1 脆弱性報告

セキュリティ上の問題を発見した場合は、公開 Issue ではなく担当者に直接報告する。

### 6.2 インシデント発生時の対応フロー

```
1. 検知・報告
2. 影響範囲の特定
3. 緊急対応（必要であればサービス停止）
4. 修正・テスト
5. リリース
6. 事後レビュー・devlog への記録
```

## 7. フェーズ別セキュリティチェックリスト

### Phase 1 リリース前チェックリスト

- [ ] `npm audit` でCritical/High の脆弱性がないこと
- [ ] XSS: ノードテキストがエスケープされて描画されていること
- [ ] JSONインポート: ファイルサイズ・スキーマバリデーションが動作すること
- [ ] テキストインポート: 異常なインデント深さでクラッシュしないこと
- [ ] CSPメタタグが `index.html` に設定されていること
- [ ] `console.log` が本番コードに残っていないこと
- [ ] `eval()` / `innerHTML` への文字列代入がないこと

### Phase 2 追加チェックリスト

- [ ] 認証: JWTの有効期限・署名検証が正しく実装されていること
- [ ] 認可: 他ユーザーのマップにアクセスできないこと
- [ ] 通信: HTTPS のみ（HTTP リダイレクト設定）
- [ ] Cookie: `HttpOnly`, `Secure`, `SameSite=Strict` が設定されていること
