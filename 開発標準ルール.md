# 開発標準ルール

**バージョン**: 1.0  
**作成日**: 2026-04-11  
**適用プロジェクト**: マインドマップツール（全フェーズ）

---

## 1. 概要

このドキュメントはプロジェクトにおける開発作業の標準ルールを定めます。
コードの品質・一貫性・追跡可能性を保ち、将来の改修・引き継ぎを容易にすることを目的とします。

---

```
mindmap-tool/

│
├── docs/                        # ドキュメント類
│   ├── adr/                     # 意思決定記録（ADR）
│   │   └── YYYYMMDD_タイトル.md
│   └── devlog/                  # 開発ログ

    ├── App.tsx                  # ルートコンポーネント

    ├── hooks/                   # カスタムフック
    ├── utils/                   # ユーティリティ関数

```

### ファイル命名規則

| 種別 | 命名規則 | 例 |
|------|---------|-----|
| コンポーネント | PascalCase | `MindMapNode.tsx` |
| カスタムフック | camelCase（use プレフィックス） | `useMindMap.ts` |
| ユーティリティ | camelCase | `exportJSON.ts` |
| 型定義ファイル | camelCase | `mindmap.ts` |
| テストファイル | 対象と同名 + `.test` | `exportJSON.test.ts` |
| ADR | `YYYYMMDD_タイトル.md` | `20260411_ライブラリ選定.md` |
| 開発ログ | `YYYYMMDD_内容.md` | `20260411_undo_redo実装.md` |

## 3. コーディング規約

### 3.1 言語・バージョン

- 言語: TypeScript（strict モード必須）
- React: 関数コンポーネント + Hooks のみ使用（クラスコンポーネント禁止）
- `any` 型の使用は原則禁止（やむを得ない場合は `// TODO: 型定義` コメントを付ける）

### 3.2 コンポーネント設計

- 1コンポーネント1ファイルを基本とする
- Props の型定義は同ファイルの先頭で `interface` として定義する
- ビジネスロジックはカスタムフックに分離し、コンポーネントはUIのみを担当する

```typescript
// Good: Props型定義 + ロジック分離
interface MindMapNodeProps {
  node: MindMapNode;
  isSelected: boolean;
  onSelect: (id: string) => void;
}

export const MindMapNode: React.FC<MindMapNodeProps> = ({ node, isSelected, onSelect }) => {
  // UIのみ
};
```

### 3.3 スタイル規約

- スタイル値は **必ず DESIGN.md で定義した CSS カスタムプロパティを使用する**
- コンポーネント内にカラーコード・px 値・font-size 値をハードコードしない

```typescript
// Bad
<div style={{ backgroundColor: '#3B82F6', padding: '8px' }}>

// Good
<div style={{ backgroundColor: 'var(--color-bg-node-root)', padding: 'var(--spacing-2)' }}>
```

### 3.4 状態管理

- ローカル状態: `useState`
- 複雑なノード操作ロジック: `useReducer` + カスタムフック
- グローバル状態: Context API（Phase 1 では外部状態管理ライブラリ不使用）

### 3.5 エラーハンドリング

- ファイルインポート時は必ずバリデーションを行い、不正データは `throw` で拒否する
- ユーザーに影響するエラーはトースト通知で表示する
- コンソールへの `console.error` 出力はエラー種別とコンテキストを明記する

```typescript
// Good
console.error('[importJSON] バリデーションエラー:', error, { input: rawData });
```

## 4. Git 運用ルール

### 4.1 ブランチ戦略

```
main          ← リリース済みコード（直接コミット禁止）
develop       ← 開発統合ブランチ
feature/*     ← 機能開発（例: feature/undo-redo）
fix/*         ← バグ修正（例: fix/node-delete-crash）
docs/*        ← ドキュメントのみの変更
```

### 4.2 コミットメッセージ規約（Conventional Commits）

```
<type>(<scope>): <日本語の説明>

[本文: 変更の理由・詳細（任意）]

[フッター: Breaking Change, Issue 番号等（任意）]
```

**type 一覧**

| type | 用途 |
|------|------|
| `feat` | 新機能追加 |
| `fix` | バグ修正 |
| `refactor` | 動作変更を伴わないリファクタリング |
| `style` | スタイル変更（DESIGN.md 反映含む） |
| `docs` | ドキュメント変更のみ |
| `test` | テスト追加・修正 |
| `chore` | ビルド設定・依存関係の更新等 |

**コミット例**

```
feat(node): Tab キーで子ノードを追加する機能を実装

ノードが選択中に Tab を押すと子ノードが追加され、
即座に編集モードに入るようにした。
```

### 4.3 プルリクエスト（PR）ルール

- 1 PR = 1 機能 or 1 バグ修正
- PR 説明に「変更内容」「テスト方法」「スクリーンショット（UI変更時）」を記載する
- セルフレビューを実施してから PR を出す

## 5. 開発ログ（devlog）記録ルール

### 5.1 目的

実装時に検討したこと・試行錯誤・判断の根拠を記録し、  
将来の改修や引き継ぎ時に「なぜそうなっているか」を追跡できるようにする。

### 5.2 記録するタイミング

- 新しい機能を実装したとき
- 技術的な選択肢を検討して何かを選んだとき
- バグの原因調査と修正を行ったとき
- 仕様の解釈について判断を下したとき

### 5.3 保存場所

`docs/devlog/YYYYMMDD_タイトル.md`

### 5.4 ログフォーマット

```markdown
# 開発ログ: [タイトル]

**日付**: YYYY-MM-DD  
**担当**: [名前またはハンドル]  
**関連機能**: [機能名・PR番号等]

---

## 背景・課題

[なぜこの作業が必要になったか]

## 検討した選択肢

### 選択肢 A: [名称]
- 概要: ...
- メリット: ...
- デメリット: ...

### 選択肢 B: [名称]
- 概要: ...
- メリット: ...
- デメリット: ...

## 決定した内容とその理由

[何を選んだか、なぜか]

## 実装の概要

[どのように実装したか（コードの構造・方針）]

## 試行錯誤・ハマったこと

[問題になったこと、試みて失敗したこと、解決方法]

## 今後の課題・TODO

[完了しなかったこと、将来改善すべきこと]
```

### 5.5 開発ログ記録の例

```
docs/devlog/
├── 20260411_ライブラリ選定.md
├── 20260415_undo_redo実装.md
└── 20260420_インデントテキストパース.md
```

## 6. ADR（アーキテクチャ決定記録）

### 6.1 ADR とは

長期的な影響を持つ技術的意思決定を記録するドキュメント。  
devlog が作業ログなのに対し、ADR はアーキテクチャレベルの「なぜそうなっているか」を残す。

### 6.2 保存場所

`docs/adr/YYYYMMDD_タイトル.md`

### 6.3 ADR フォーマット

```markdown
# ADR: [タイトル]

**日付**: YYYY-MM-DD  
**ステータス**: 提案中 / 承認済み / 廃止済み

## 背景

[この決定が必要になった状況]

## 決定内容

[何を採用するか]

## 理由

[なぜこの選択をしたか]

## 代替案

[検討したが採用しなかった選択肢]

## 影響・トレードオフ

[この決定によって生じる制約やトレードオフ]
```

## 7. テスト規約

### 7.1 テスト対象

| 対象 | テスト種別 | 優先度 |
|------|---------|--------|
| ユーティリティ関数（export/import/parse） | ユニットテスト | 必須 |
| カスタムフック（useMindMap, useHistory） | ユニットテスト | 必須 |
| コンポーネント | スナップショット or インタラクションテスト | 推奨 |
| E2E（ユーザー操作フロー） | Playwright | Phase 2 以降 |

### 7.2 テストファイル配置

テスト対象ファイルと同じディレクトリに置く。

```
src/utils/exportJSON.ts
src/utils/exportJSON.test.ts   ← 同階層
```

## 8. 依存ライブラリ管理

- `package.json` の `dependencies` / `devDependencies` を明確に分ける
- ライブラリ追加時は ADR または devlog に選定理由を記録する
- セキュリティアップデートは速やかに対応する（`npm audit` を定期実行）
- 不要なライブラリは都度削除する

---

## 9. 禁止事項

| # | 禁止 | 理由 |
|---|------|------|
| 1 | CSS カスタムプロパティ以外での色・サイズ指定 | DESIGN.md による再現性確保のため |
| 2 | `any` 型の無断使用 | 型安全性の維持 |
| 3 | `console.log` の本番コードへの混入 | ログの汚染防止 |
| 4 | `eval()` / `innerHTML` への文字列連結代入 | XSS 脆弱性防止 |
| 5 | `main` への直接コミット | 意図しないリリース防止 |
| 6 | 外部ライブラリの追加記録なし | 将来の依存関係管理のため |
